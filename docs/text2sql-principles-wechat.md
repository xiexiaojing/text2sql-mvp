## 一、先搞懂几个基础概念

后面会反复用到这些词，先对齐含义。

### 1. 什么是「召回」

**召回**（Retrieval），在 AI 搜索/匹配语境里，意思是：**从大量候选项里，先把「可能相关」的一小批找出来**。

> 注意英文别混：**Retrieval** 是召回这个动作/阶段；**Recall** 通常是评估指标「召回率」（该捞的里实际捞到了多少）。下文说的召回，一律指 Retrieval。

可以类比图书馆找书：

- 馆里有 10 万本书（全部业务意图）
- 你问管理员「有没有讲支付渠道分析的」
- 管理员不会一本本翻，而是先按主题索引**召回**（Retrieval）出 5～10 本可能相关的
- 再从中**精排**（Ranking / Reranking），挑出最匹配的那一本

Text2SQL 里，召回发生在意图识别阶段：用户问一句自然语言，系统要从几十个业务意图里，先找出 3～6 个「可能命中」的候选，再进一步确定是哪一个。

**召回 ≠ 最终答案。** 它只是缩小范围的第一步。最终选哪个意图、填什么参数、生成什么 SQL，是后面的事。

### 2. 什么是「向量」

**向量**（Vector），就是一串数字，例如 `[0.12, -0.34, 0.56, …]`，通常有几百到几千维。

在 NLP（自然语言处理）里，文本会先被转成向量——语义相近的文本，向量在空间里距离更近。比如：

- 「各支付渠道交易金额分布」
- 「支付渠道金额占比饼图」

这两句字面不同，但向量距离很近，系统会认为它们「意思差不多」。

### 3. 什么是 Embedding

**Embedding**（嵌入），是把文本映射成向量的过程，或承担这个任务的模型。

你可以把它理解成一种「语义坐标系」：问句进去，出来一个坐标；坐标近，语义就近。Text2SQL 里用 Embedding 做向量召回，**只负责把文字变成可比较的坐标，不负责写 SQL**。

### 4. 什么是 RAG

**RAG**（Retrieval-Augmented Generation，检索增强生成），完整流程是：

```text
用户问题 → 检索（Retrieval）相关资料 → 把资料拼进 Prompt → LLM 自由生成答案
```

关键词是 **Generation（生成）**——检索到的内容，最终交给 LLM 自由发挥。

Text2SQL 快速模式**只借用了 RAG 的前半段「检索」**，后半段不交给 LLM 自由写 SQL，而是走预定义的 SQL 模板。所以：**向量召回和 RAG 是亲戚，但不是一回事。**

### 5. 什么是「意图」和「槽位」

- **意图**（Intent）：用户想干什么。比如「查支付渠道金额分布」是一个意图，「查退款趋势」是另一个。
- **槽位**（Slot）：意图里需要填的参数。比如「近 7 天每日退款笔数趋势」里，`近7天` 是时间槽位，`退款` 是业务对象槽位。

意图识别 = 判断是哪种业务问题；槽位抽取 = 从问句里抠出具体参数。

### 6. 什么是「模板」

**模板**（Template），是预先写好、经过审核的 SQL 骨架，用 `{{slot_name}}` 占位，运行时填入参数。

比如「按渠道统计金额」的模板可能是：

```sql
SELECT channel, SUM(amount) AS total
FROM payment_order
WHERE tenant_id = {{tenant_id}}
GROUP BY channel
```

用户问法千变万化，但 SQL 逻辑固定——这就是模板的价值：**口径一致、可测试、可审计。**


## 二、向量召回：意图识别的第一步

上一篇 [《Text2SQL 入门：快速模式与专家模式，用支付 Demo 讲清原理》](https://github.com/xiexiaojing/text2sql-mvp/blob/main/docs/text2sql-intro-wechat.md) 里讲到，意图识别的第一个重要过程是**向量召回**。那什么是向量召回？它和 RAG 有什么关系？如果走 LLM，这个阶段 LLM 本质上做了什么？

### 1. 向量召回怎么做

在 `business_semantics.yaml` 里，每个意图除了关键词，还配置了 `semantic.queries`——一组「代表这个意图」的语义问句。例如「支付渠道金额占比」「各渠道交易金额分布图」。

系统启动时，这些问句会被 Embedding 成向量，建索引。用户问题进来，同样转成向量，然后做**余弦相似度**比较——算两个向量夹角，距离越小越相似——取出 top-k 个最近的意图作为候选。

**全程不调用 LLM**，只调 Embedding API（或本地向量器做离线测试），延迟通常在毫秒级。

### 2. 为什么需要向量召回

纯关键词匹配快、稳，但扛不住说法变化：

| 配置里的关键词 | 用户实际问法 |
|---|---|
| 「各支付渠道交易金额分布」 | 「微信支付宝各占多少」 |
| 「按渠道统计」 | 「渠道占比饼图」 |

向量召回靠语义相似度，能兜住这类变体——**不用穷举所有问法，也不用每来一个新说法就改 Prompt。**

但它也有边界：语义像，不代表业务口径对。所以向量召回之后，还要过关键词规则、边界样本过滤，必要时才请 LLM 精排。

### 3. 向量召回 vs RAG：借了一半，没借全部

| | 完整 RAG | Text2SQL 向量召回 |
|---|---|---|
| 第一步 | 向量检索（Retrieval）相关内容 | 向量检索相关意图 |
| 检索对象 | 文档、Schema、知识片段 | 预置的业务语义问句 |
| 检索之后 | LLM 自由生成 | 规则打分 → 模板渲染 SQL |
| 输出是否可控 | 概率性，需护栏 | 命中模板则完全确定 |

行业里常见的「Schema 向量召回 + LLM 写 SQL」，是把 RAG 走到底。我们的快速模式是 **Retrieve → Rank → Template**——只借检索，生成阶段不自由发挥。

### 4. 如果这个阶段走 LLM，它本质上在做什么

向量召回给出 top-k 候选后，仍可能有两个意图距离接近，或需要从问句里抠参数。这时才**可选地**请 LLM 出场——在 text2sql-mvp 里叫 **LLM 精排 + 槽位抽取**。

LLM 在这里**不是**对着全库 Schema 写 SQL，而是在**已经缩小的候选集里做受限选择**（constrained selection）：

- **输入**：用户问题 + 3～6 个候选意图的结构化描述
- **输出**：JSON——选哪个 `intent_id`、置信度、槽位字典
- **约束**：`intent_id` 必须在候选列表里；槽位只能填该意图允许的字段

本质上做两件事：

1. **分类/精排**：多个语义相近的意图里，选最匹配的一个；
2. **信息抽取**：从自然语言里提取结构化参数（类似 NER + slot filling）。

完整链路：

```text
用户问题
  → 向量召回（Embedding，不用 LLM）
  → 关键词/正则规则打分（不用 LLM）
  → 多个候选距离接近 → LLM 精排 + 抽槽（最多一次）
  → 命中意图 → SQL 模板渲染 → 执行
```

很多问法向量 + 规则就能定下来，**零次 LLM**；只有边界 case 才消耗 Token。

### 5. 向量召回的工程保险

text2sql-mvp 的向量索引还有几层过滤，防止「语义像但业务不对」：

- **negative_queries**：语义相近但属于别的意图的问句——距离太近则排除；
- **boundary_queries**：意图边界样本，防止 A/B 两个意图互相抢；
- **lexical override**：关键词规则置信度极高时，直接覆盖向量结果。

向量召回解决「说法变化」，规则解决「业务边界」——两者配合，不是二选一。


## 三、模板怎样泛化

### 1. 什么是泛化

**泛化**（Generalization），在工程语境里，指：**把多个具体、重复的配置，抽象成更通用、可复用的结构**，而不是每来一个场景就堆一个新意图。

不是让一个模板回答所有问题，而是当相似场景积累到一定程度，合并成参数化模板。

比如一开始你可能有三个意图：

- 「支付订单按渠道统计」
- 「支付订单按状态统计」
- 「退款按日趋势」

它们的 SQL 结构类似：某表 + GROUP BY 某维度 + COUNT/SUM + 时间过滤 + 租户隔离。继续堆第四个、第五个，维护成本会指数上升。泛化的方向是抽出**参数化模板**——`group_by_column`、`metric`、`time_range` 变成槽位，一个模板覆盖多类分组统计。

text2sql-mvp 里每个意图对应一条 SQL 模板，模板里用 `{{slot_name}}` 占位，运行时注入参数。泛化就是**减少模板数量、增加槽位维度**，让配置结构更抽象。

新增场景达到一定程度，要让 AI 重构，让模板更通用。保存历史所有调试过的问题，做成自动化测试用例，保证对已经调试过的场景效果不退化。

### 2. 什么时候该泛化，什么时候加新意图

| 情况 | 做法 |
|------|------|
| 问法不同，SQL 逻辑相同 | 给现有意图补 keywords / semantic.queries |
| SQL 逻辑相同，只是分组维度不同 | 考虑合并意图，增加 slot |
| 统计口径不同（不同表、不同 WHERE、不同聚合） | 新增意图 + 新模板 |
| 高频场景，规则能覆盖 | 优先加 match 规则，不必泛化 |

**99% 的查询重复性极高**——支付 Demo 里 8 个意图就覆盖了绝大部分高频问数。不要为了「泛化而泛化」；当 eval_cases 里同类意图超过 5 个、SQL diff 高度相似，才是重构信号。

### 3. 用测试锁住已有效果

泛化或重构最怕「改 A 坏 B」。text2sql-mvp 的做法：

1. **eval_cases/cases.yaml**——每条典型问句是一个回归用例，断言命中的表、特征（group_by、domain_filter 等）；
2. **调试过程中踩过的坑**——用户真实问法、边界 case，全部沉淀为测试；
3. **CI 跑一遍**——改配置文件后自动验证，效果不退化才合并。

这是**配置驱动系统的安全网**。上层 Prompt 可以天天改，底层模板和意图映射必须有回归保护。

### 4. 泛化的边界

探索性问法、跨表复杂 JOIN、一句话生成报告——不适合硬塞进模板，应走**专家模式（Agent OS）**或 Schema 驱动的 LLM 兜底路径（框架里保留，默认不开启）。

快速模式的核心假设是：**常见业务口径可枚举、可审核、可模板化。** 超出假设，诚实拒绝或切换架构，比强行泛化更靠谱。


## 四、LLM 语言精修：开始和结束那两次调用

很多人在做工程的时候，会在开始和结束时都调用一次 LLM。在开始时主要判断路由、意图识别；在结束时对返回结果进行友好的封装。这个其实不用 LLM 也能做。

### 1. 开始：路由与意图识别

第二节已说：向量召回 + 关键词规则 + 可选 LLM 精排。多轮追问改写也一样——text2sql-mvp 用**对话历史 + 规则**把「折线图也生成一下」改写成完整问句，不调 LLM。

```text
第一轮：「各支付渠道交易金额分布饼图」
第二轮：「折线图也生成一下」
         ↓ 规则改写
         「各支付渠道交易金额分布折线图」
```

规则改写依赖上一轮命中的意图和主题，不是模型「记忆力好」。这比 LLM 改写更快、更稳、更可审计。

### 2. 结束：结果封装

查库完成后，快速模式走 **ResultFormatter + 图表模块**，全程不用 LLM：

- **自然语言摘要**：根据结果形状（单行计数、分组统计、空结果）套模板句式——「查询结果为 128」「支付渠道分布如下」；
- **表格列名**：从 `business_semantics.yaml` 和 schema 映射中文标签；
- **图表类型**：问句里检测「饼图」「折线图」「柱状图」等关键词，或按意图默认值；
- **ECharts 配置**：按意图 + 图表类型 + 数据列名，规则生成 JSON。

```text
查库结果 + 意图 + 问句关键词
  → ResultFormatter 套模板出 answer
  → visualization 模块出 echartsOption
  → 前端渲染
```

LLM 做结果封装，往往只是把「128」润色成「共有 128 笔订单」——信息零增量，延迟 +1～3 秒，还不稳定。工程上**模板足够**。

### 3. 什么时候才需要 LLM 精修

| 场景 | 是否用 LLM |
|------|-----------|
| 快速模式：高频统计问数 | 不需要 |
| 快速模式：边界意图、槽位模糊 | 意图识别阶段最多一次 |
| 专家模式：跨系统报告、复杂研判 | 需要，走 Agent 架构 |
| 空结果、异常口径的友好提示 | 规则 + 意图级模板即可 |

**LLM 语言精修适合「内容本身需要推理和创作」的场景**——比如「分析本月退款异常原因并给出建议」。这是专家模式的事，不是快速模式该承担的。


## 五、底层原理就这几块

把 Text2SQL 快速模式和常见 AI 技术对齐，底层其实就五层：

```text
┌─────────────────────────────────────────────────┐
│  1. 向量召回（Embedding 相似度，RAG 的检索段）    │
├─────────────────────────────────────────────────┤
│  2. 启发式匹配（关键词/正则，确定性兜底）          │
├─────────────────────────────────────────────────┤
│  3. LLM 精排（可选，受限选择，不是自由生成）      │
├─────────────────────────────────────────────────┤
│  4. 模板渲染 + SQL 护栏（Harness，不用 LLM）      │
├─────────────────────────────────────────────────┤
│  5. 结果组装（Formatter + 图表规则，不用 LLM）   │
└─────────────────────────────────────────────────┘
```

上层应用天天变：换模型、换 Agent 框架、换 UI。但这五层分工不变——**检索（Retrieval）缩小范围，规则保证确定，LLM 只补推理，Harness 保准确。**

和「端到端 LLM」的本质区别不在「用不用 AI」，而在**生成阶段是否自由发挥**。RAG 检索 + LLM 自由写 SQL，是另一条路；快速模式是 RAG 的检索 + 模板生成，LLM 调用上限一次，很多问法零次。


## 六、结语

AI 技术迭代快，容易让人以为每个新场景都要新架构。实际上 Text2SQL 快速模式的底层就这些：

- **召回（Retrieval）**——从大量候选项里先找出可能相关的一小批；
- **向量召回**——用 Embedding 按语义相似度做这件事；
- **RAG**——检索 + LLM 自由生成；我们只用前半段；
- **模板泛化**——配置抽象 + 回归测试，而不是无限堆意图；
- **语言精修**——开始和结束都能用规则，LLM 不是默认选项。

搞懂这些概念，换社区治理、换支付、换零售，改的是 `business_semantics.yaml` 里的业务口径，不是架构本身。
