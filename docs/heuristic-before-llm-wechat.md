Text2SQL 分意图识别、SQL 执行、结果返回三阶段，多数框架前后都强依赖 LLM，慢且不稳。更优路径是规则先兜 80% 高频场景，定不下来再请模型。启发式可控可测，LLM 灵活但有延迟和不确定性——工程上两者协同，不是二选一。

Text2SQL 意图识别为什么要少用大模型


## 一、从「准确、快速」出发

我在开源 Text2SQL 框架 [text2sql-mvp](https://github.com/xiexiaojing/text2sql-mvp) 里有一条核心原则：**尽量减少对大模型（LLM）的依赖**。

LLM，即大语言模型——ChatGPT、DeepSeek 这类模型，你发一段文字，它返回一段文字，背后靠海量参数做语义理解。

Text2SQL 的整体流程可以拆成三个阶段：

| 阶段 | 做什么 | 常见做法 |
|------|--------|----------|
| **意图识别** | 听懂用户问的是哪类业务问题 | 多数框架：直接调 LLM |
| **SQL 执行器** | 生成 SQL、校验、执行、护栏 | 相对成熟，规则 + 模板为主 |
| **返回结果** | 组织答案、选图表形态 | 多数框架：再调一次 LLM |

问题在于：**意图识别**和**返回结果**这两步，很多实现几乎步步调 LLM。每多一次调用，延迟就从毫秒级跳到秒级，成本成倍上涨，结果还可能「今天对、明天错」。

我们的思路是反过来的：**能不用 LLM 就不用，实在搞不定再请它出场。**


## 二、一个真实场景：AI 原生 vs AI 增强

LLM 在意图识别上确实更「聪明」。这一点在我做的 AI 助理工具里体会很深——我们同时用了两种模式，名字容易搞混，先对齐含义：

### AI 原生

**AI 原生**，指走 **为 AI 专门设计的原生通道**——对话框直连 Text2SQL、Agent OS 等后端能力，不模拟人工点页面。

比如用户问「各支付渠道交易金额分布，用饼图展示」，系统识别意图后直接查库、返回图表，全程走结构化 API，**更快、更准、更可审计**。这是我们产品里「AI 助理工作台 / 快速模式」的主路径。

### AI 增强

**AI 增强**，指在 **已有 Web 业务系统上做增强**，而不是从零写一条 AI 专用链路。常见两种形态：

1. **直达页面**：规则判断出目标业务页（统计表单、台账查询等），生成筛选参数后直接打开对应界面——相当于帮用户「填好条件、跳到对的页面」。
2. **GUI Agent**（Graphical User Interface Agent，图形界面智能体）：识别不了具体页面时，让 AI 像真人一样在界面上点按钮、填表单、切 Tab——把「看屏幕 + 点鼠标 + 敲键盘」交给 Agent 完成。

增强模式解决的是：**平台里已经有一堆成熟页面，怎么用 AI 让用户少点几步、少找几个菜单。**

### 核心问题：AI 怎么知道该走哪条路？

同一个对话框里，用户说「帮我查一下各渠道交易金额」——系统要先判断：

- 能不能走 **AI 原生**（Text2SQL 直接查数）？
- 还是要走 **AI 增强**（打开统计页 / 让 GUI Agent 去页面上操作）？

本质上，这就是**意图识别**。

两种做法：

| 方式 | 优点 | 缺点 |
|------|------|------|
| **规则匹配**（正则、关键词） | 快（毫秒级）、稳、可控、零 Token 成本 | 规则外的问法可能漏掉 |
| **LLM 判断** | 灵活，能识别更多变体说法 | 慢（秒级）、费 Token、不稳定 |

LLM 判断则相反：同一个问题，多一个逗号、少一个句号，有时就会给出不同答案。这不是偶发 bug，而是概率模型的本质特性。


## 三、更理想的方案：规则先行，模型兜底

工程上更稳妥的路径是：

```text
用户输入
  → 先用正则 / 关键词 / 逻辑规则匹配
  → 命中 Text2SQL 类意图 → 走 AI 原生（直连查库，快、稳）
  → 命中某业务页面 → 走 AI 增强（直达页面）
  → 都未命中 → 再用 LLM 兜底判断
  → 仍不确定 → 走 AI 增强（GUI Agent 操作页面）
```

如果规则能兜住 **80%** 的高频场景，整体速度和成本都会显著优于「一上来就调 LLM」。

而且有个很实用的经验：**当某个问法反复漏判时，与其花半天调 Prompt，不如加一条匹配规则**——直接、稳定、可测试、可回归。

举个例子：用户问「渠道占比饼图」——如果现有规则只覆盖了「各支付渠道交易金额分布」，这条就可能漏掉。补一条关键词或正则匹配，比改 LLM 提示词靠谱得多。


## 四、Bitter Lesson 说的，和工程落地不矛盾

有人可能会想：Google 研究员 Rich Sutton 提出的 **Bitter Lesson**（苦涩的教训）告诉我们——**算力和通用方法最终会碾压人工设计的规则**，很多现在搞不定的事，以后模型自己会搞定。那还有必要写规则吗？

这里需要澄清一个概念。

### 什么叫「启发式」？

**启发式**（Heuristic）：基于经验法则的、**不保证最优解**但追求**快速有效**的解题方法。

在意图识别里，启发式特指：**人工预定义的确定性规则**——正则表达式、关键词列表、if-else 逻辑判断——用来快速、稳定地对用户输入分类，**不依赖**统计模型或数据训练。

它和 LLM 不是同一个维度的东西：启发式是「写死的逻辑」，LLM 是「概率性的推理」。

### 但以下几件事，LLM「什么时候能稳定搞定」的期限是不可预估的

1. **响应时间可控**——业务系统需要 P99 在 1 秒内，LLM 单次调用通常 1～5 秒，多步串联更慢
2. **成本可预算**——每条查询花多少 Token、多少钱，LLM 路径难以精确预估
3. **结果可复现**——同样的问题，今天和明天答案一致，审计和排错才做得到
4. **错误可定位**——规则命中了哪条、为什么走这条路径，日志一目了然；LLM 黑盒出错，很难查
5. **离线 / 降级可用**——模型 API 挂了或限流，规则路径仍可服务核心功能

Bitter Lesson 说的是**长期研究方向**；工程落地面对的是**今天就要上线的 SLA**（Service Level Agreement，服务等级协议）。两者不矛盾。

### 正确的分工

```text
高置信度、高频场景  →  启发式快速兜底（正则、关键词、模板）
长尾、模糊、复杂语义  →  交给 LLM
```

**两者并非互斥，而是互补协同。**

就像 Text2SQL 框架里的意图识别：

```text
向量语义召回（相似度匹配，不用 LLM）
  → 关键词 / 正则规则打分（不用 LLM）
  → 仍定不下来，LLM 精排兜底（最多一次）
  → 命中意图 → SQL 模板渲染 → 规则校验 → 执行
  → 结果展示走模板匹配（表格 / 饼图 / 折线图，不用 LLM）
```

LLM 负责**搞不定的推理**；Harness（执行框架、规则、模板、护栏）负责**现实世界的准确和快速**。


## 五、三条落地原则

**第一，先写规则，再调 Prompt。**

漏判一个高频问法，加一条匹配规则比改提示词快，效果也稳定。

**第二，规则要有测试回归。**

每条规则配上典型问句，CI 里跑一遍，防止改 A 坏 B。

**第三，LLM 是兜底，不是默认路径。**

能零次 LLM 命中的问法，就不要为了「更智能」硬调一次。用户感知到的是速度，不是背后调了几个模型。


## 六、结语

「全链路 LLM」听起来先进，上线后往往是：**慢、贵、难排错、口径飘。**

不是不用大模型，而是**把大模型用在刀刃上**——规则和模板兜住八成高频场景；搞不定的意图，再请 LLM 出场。

**先规则、后模型。** 这是 Text2SQL 快速模式能稳定在约 1 秒内响应、口径可审计的工程底气。
