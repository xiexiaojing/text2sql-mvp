为什么不用一个大模型端到端？

VIP 课结束后，有同学提了一个很典型的问题：

「我看很多 Text2SQL 的实现，从意图识别、改写增强、向量库召回后的 Schema 召回、可行性评估、语义检查到 SQL 生成，大面积依赖 LLM，所以确实很慢。你们昨天的设计思路不太一样，颇受启发。」

这篇文章就把这个问题说清楚：为什么我们不走「一个大模型端到端」的路。


一、两种 Text2SQL，两种产品假设

行业里常见的 Text2SQL 流水线，大致长这样：

```text
用户问题
  → LLM 意图识别
  → LLM 改写增强
  → 向量库召回 Schema
  → LLM 可行性评估
  → LLM 语义检查
  → LLM 生成 SQL
  → 执行
```

每一步都「更智能一点」，串起来却有一个通病：**慢，而且不稳定。**

我知道有大厂做 Text2SQL，一个查询平均 **2 分钟**。里面大量的语言检查、护栏、知识库调用，只能做成内部员工使用的产品——普通用户等不了。

我们的快速模式（Text2SQL 框架）走的是另一条路：

```text
用户问题
  → 前端规则改写（多轮追问，不用 LLM）
  → 向量召回 + 启发式匹配（不用 LLM）
  → 实在定不下来，才 LLM 兜底（最多一次）
  → 模板渲染 SQL → 安全校验 → 执行
  → 模板匹配输出形态（表格/饼图/折线图，不用 LLM）
```

目标很明确：处理 **80% 以上高频、简单的统计问数**，响应约 **1 秒**，口径稳定，可测试，可审计。

剩下 20% 的复杂任务——跨系统编排、复杂研判、一句话生成报告——交给 **专家模式（Agent OS）**，允许多次调用大模型，走另一套架构。

一句话：**Text2SQL 负责常见问数，Agent 负责复杂任务。** 互补，不是二选一。


二、为什么不用一个大模型端到端？

先承认一点：**纯关键词扛不住说法变化，纯 LLM 又贵又不稳。** 「先召回、再精排」本身是合理架构。

问题出在：**不能做成串行调用堆叠。**

前面一步错了，后面再「检查」「评估」，往往是在 **确认错误**，不是纠正错误。你堆 6 步 LLM，不等于 6 倍准确，常常是 6 倍延迟、6 倍成本，错误还在前面就埋下了。

所以很多步骤，其实可以用 **规则、配置、向量、模板** 替代——又快又准：

| 环节 | 常见做法 | 快速模式做法 |
|------|----------|--------------|
| 多轮追问改写 | LLM 理解上下文 | 前端规则 + 对话历史 |
| 意图识别 | LLM 直接分类 | 向量召回 → 正则/关键词 → LLM 兜底 |
| SQL 生成 | LLM 对着全库 Schema 写 | 配置文件里的 SQL 模板 |
| 结果展示 | LLM 组织语言 | 模板匹配图表类型 |
| 安全护栏 | LLM 自检 | 白名单表字段 + 只读策略 + 租户注入 |

**LLM 负责推理，Harness 负责现实。** 该用模型的地方用，不该用的地方别硬塞。

如果是项目刚开始，建议 **做极简架构，再按需添加**。每加一步 LLM 调用，往往意味着复杂度上升、速度变慢。加功能容易，加回去难。


三、课后三个问题，集中回答

**Q1：快速模式是不是只有意图识别用 LLM？输出也不走 LLM 吗？**

是的，理解要精确一点：

- **意图识别**：向量召回 + 启发式匹配为主；只有两者定不下来时，LLM 才兜底。
- **SQL 生成**：走配置文件里的模板，不走 LLM。
- **结果输出**：走模板匹配前端形态（表格、饼图、折线图等），不走 LLM。

所以快速模式的 LLM 调用上限是 **一次**，很多问法 **零次** 就能命中。

**Q2：如果要出一份数据报告，快速模式支持吗？**

**不支持。** 报告属于复杂任务，要走 **专家模式**。

快速模式和专家模式是 **两种架构**。会话开始后 **不能切换**——会涉及上下文中断。DeepSeek 等厂商做产品分层，多半也有类似考虑：快的事快做，难的事另开一条路。

**Q3：规则匹配是不是要建立大量样本库？模板放向量库里吗？**

要建，但 **量不一定很大。**

一个「意图」应对的不是某一个问句，而是 **一类场景**，可以继续抽象。以支付领域为例，用户高频查询的场景其实很少——之前有句话：99% 的设计是为了应对 1% 的场景；反过来说，**99% 的查询重复性极高**，快速模式就是为这个准备的。

模板放进向量库是一种方式，但不是唯一方式。YAML 配置文件、关系数据库、搜索引擎都能做，关键看场景。了解了本质——**把业务口径映射成可审核的模板**——剩下的还是传统架构能力：配置管理、测试回归、权限隔离。


四、本体论：别让大模型猜你的业务

还有一个相关话题：为什么光有 LLM 不够？

缺乏业务定义时，大模型容易「幻觉」。比如问「社区有多少人」，业务方要的是 **工作人员总数**，模型却理解成 **居民总数**——口径错了，SQL 再漂亮也没用。

**本体论** 要解决的，就是这个「语义鸿沟」：统一业务语言，定义实体、关系和规则，让系统理解的是 **业务方的真实意图**，不是模型的自由发挥。

知识图谱是本体的一种实现形式；构建工具可以用 Protégé，术语参考 Schema.org。向量库里存知识图谱，先检索再精排——这条路和 Text2SQL 快速模式的思路一致：**先缩小范围，再动用大模型。**


五、落地上的几句实话

- **试验阶段**：Python 生态好，开发效率高，降低试错门槛。
- **生产阶段**：高并发、可维护性上，主流工程栈仍是 Java、Go——AI 能力可以嵌进去，不必全部重写。
- **起步原则**：极简架构先行，别一上来就堆 6 步 LLM 流水线。能用一个模板解决的，别用六个模型讨论。

同学说「对落地还有点模糊，再研究一下」——这很正常。建议从一个具体场景跑通，比如「各支付渠道交易金额分布」，对照开源 Demo 看配置文件怎么写，比先啃完整架构容易得多。

六、结语

「全链路 LLM」听起来先进，上线后往往是：**慢、贵、难排错、口径飘。**

不是不用大模型，而是 **把大模型用在刀刃上**——搞不定的意图再请它出场；SQL、护栏、展示，交给配置和模板。

快速模式保速度和稳定，专家模式保复杂和灵活。两套架构，各干各的，会话内不混用——这是产品分层，也是工程诚实。
