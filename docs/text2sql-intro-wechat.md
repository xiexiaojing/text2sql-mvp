Text2SQL 入门：快速模式与专家模式，用支付 Demo 讲清原理

本文配套开源项目 text2sql-mvp，用支付/交易 Demo 说明 Text2SQL 原理与开发方式，不涉及具体公司业务，也不绑定任何 Agent 平台。

开源地址：https://github.com/xiexiaojing/text2sql-mvp

克隆仓库后执行 start.sh，浏览器打开本地 8777 端口的 chat 页面即可体验。


一、产品怎么分：快速模式 vs 专家模式

在实际产品里，我们把问答能力拆成两档。

快速模式，走 Text2SQL。目标是解决 80% 以上用户最高频的统计类问题——订单有多少、按渠道分布、退款趋势、商户排名这类。要求快、准、口径稳定，用户问完几秒内就有结果。

专家模式，走 Agent OS。留给剩下 20% 的高难度任务——态势感知、跨系统编排、复杂研判、需要多步推理和工具链协作的场景。这类问题不适合硬塞进 SQL 模板，交给 Agent 更合适。

一句话：Text2SQL 负责「常见问数」，Agent 负责「复杂任务」。两者互补，不是二选一。


二、Text2SQL 在干什么？

用户问一句：「各支付渠道交易金额分布，用饼图展示」。

系统要做三件事：听懂问题、查库、展示结果。

听懂，就是知道要按渠道分组、汇总金额、用饼图展示。

查库，就是生成并执行一条安全的只读查询。

展示，就是返回文字说明，加上表格或图表。

很多 Demo 的做法，是把整张数据库表结构丢给大模型，让它直接写 SQL。演示能跑，但上线容易出问题：查错租户、扫全表、口径不一致、敏感字段泄露。

本框架的主路径不是这条路。更稳妥的做法是：先匹配业务口径，再生成 SQL，最后过安全检查。全库 Schema 喂给大模型的方式，框架里保留为可选兜底，默认不依赖它。


三、架构就三层

第一层，API 入口。接收用户问题、租户 ID、对话历史。

第二层，核心运行时。负责理解问题、生成 SQL、校验、查询、格式化结果和图表。

第三层，配置文件。业务口径、白名单表、性能阈值，都在这里。日常开发，九成时间改配置文件，代码主要负责安全执行。

主要配置文件：

business_semantics.yaml — 定义「什么问题对应哪条 SQL」

whitelist_tables.yaml — 定义「允许查哪些表、字段、表之间怎么关联」

performance.yaml — 超时、扫描行数上限等

eval_cases/cases.yaml — 回归测试用例


四、核心运行时：三件事

运行时主要做三件事：意图识别、执行引擎、组装结果。横向还有 SQL 护栏、租户隔离、审计日志等安全措施，本文不展开。

1. 意图识别（重点）

用户的问题不会直接变成 SQL，先要回答：这是哪类业务问题？

框架的做法是配置驱动，而不是让大模型对着全库 Schema 自由发挥。

每个业务意图在 business_semantics.yaml 里定义：匹配关键词、示例问句、语义问句、需要的参数（槽位）、对应的 SQL 模板。

识别过程大致是：向量语义召回，把用户问题和预置的语义问句做相似度匹配；关键词规则打分，用 match.any / match.all 做确定性匹配；可选 LLM 精排，在少量候选意图里选最准的一个并抽取参数；启发式抽槽，用正则和规则从问句里抠出渠道名、时间范围、图表类型等参数。

例如「近 7 天每日退款笔数趋势」会命中 refund_daily_trend 这类意图，而不是让模型从零写 SQL。

泛化场景怎么做？

业界常见方案：把数据库所有 Schema 喂给大模型，让它动态生成 SQL。如果表结构定义不规范、字段注释不完备，正确率往往很差，调试成本极高。

本框架的主路径是：常见高频问题，提前在配置文件里定义好意图和 SQL 模板，保证口径一致、可测试、可审计。Schema 驱动 + LLM 生成 SQL 作为兜底路径，可按需开启或关闭——适合探索性问法，但不建议作为生产主路径。

2. 执行引擎

意图确定后，渲染 SQL 模板，注入租户过滤条件，过 SQL 护栏（只允许 SELECT、白名单表字段、行数上限），再执行查询。没有配置过的统计口径，明确拒绝，而不是随便查一张表凑答案。

3. 组装结果

查库完成后，格式化成用户看得懂的形态：自然语言摘要（answer）、结构化表格（table）、需要时生成 ECharts 图表配置（echartsOption）。例如饼图、柱状图、折线图，会从问句里识别「饼图」「折线图」等关键词。


五、一条完整链路（两轮对话）

第一轮，用户问：「各支付渠道交易金额分布饼图」。

系统内部会：识别为支付渠道金额分布这类业务意图，套用预先写好的 SQL 模板，自动加上当前商户/租户的数据范围过滤，做 SQL 安全检查，执行查询，最后生成饼图。

用户看到的是：各渠道占比说明，加一张饼图。

第二轮，用户追问：「折线图也生成一下」。

这句话单独看，系统并不知道「什么数据的折线图」。框架会结合上一轮对话，改写成「各支付渠道交易金额分布折线图」，再走同样的流程。

多轮对话靠追问改写实现，不是模型「记忆力好」。


六、想支持新问题，改哪里？

场景 A：已有类似统计，只是问法不同

例如已支持「支付订单按渠道统计」，用户问「微信支付宝各占多少」。

只需在 business_semantics.yaml 里，给对应意图补充关键词和示例问句，通常不用写新 SQL。

场景 B：全新统计口径（最常见）

例如「近 7 天每日退款笔数趋势」。

第一步，在 business_semantics.yaml 新增一个意图，写清楚匹配关键词和示例问句。

第二步，在同文件的 SQL 模板区，写对应的查询逻辑。

第三步，确认相关表已在 whitelist_tables.yaml 白名单里。

第四步，在 eval_cases 加测试问句。

第五步，跑测试确认能命中、SQL 合法。

模板里不必手写租户过滤条件，运行时会自动注入。

场景 C：换图表类型

如果「折线图也生成一下」理解错了，改对话追问改写逻辑。

如果改写对了但不出图，改图表生成模块，并确认该意图支持图表。

场景 D：提示「未配置该统计口径」

说明问题没有命中任何可执行的意图。依次检查：有没有对应意图、匹配规则是否太严、示例问句是否覆盖用户说法、意图状态是否为可执行。


七、四条安全原则

第一，租户 ID 由后端注入，不让模型自己决定查哪个商户的数据。

第二，只允许 SELECT，写操作一律拒绝。

第三，只能查白名单里的表和字段，敏感列默认不可见。

第四，没有配置过的统计口径，明确拒绝，而不是随便查一张表凑答案。


八、和「直接让 AI 写 SQL」的区别

口径一致性：直接让 AI 写 SQL 不稳定；本框架用模板保证。

租户隔离：直接让 AI 写容易漏；本框架后端强制注入。

新需求：改 prompt 碰运气；改配置文件可测试。

排错：直接让 AI 写很难查；本框架有审计日志，记录命中了哪个意图、用了哪条 SQL。

适用场景：直接让 AI 写适合 Demo 和探索；本框架适合要上线的业务系统，尤其是快速模式下的高频问数。


九、开发检查清单

新增一个支付类问答能力时，可以按这个顺序：

用一句话定义统计口径（查哪张表、按什么分组、算什么指标）。

在 business_semantics.yaml 增加意图、匹配规则和示例问句。

编写 SQL 模板。

在 whitelist_tables.yaml 声明表和关联关系。

在 eval_cases 加两到三条典型问法。

跑测试通过。

需要图表时，问题里带上「饼图」「柱状图」「折线图」等词。

需要多轮时，测试「那按状态呢」「折线图也生成一下」这类追问。


十、结语

Text2SQL 不是「教会 AI 写 SQL」，而是把常见业务问题，映射成少量、可审核、可测试的查询模板，承接产品里的快速模式。

从一个具体场景跑通，比如「各支付渠道交易金额分布」，再按同样模式扩展，比一开始啃完整架构容易得多。

高难度任务交给 Agent 专家模式，两者配合，才是完整的产品形态。
