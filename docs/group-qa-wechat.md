智能问数群聊实录：模板、冷启动与兜底

前几篇讲 Text2SQL 底层原理、向量召回与「先规则后模型」之后，群里陆续抛出一批很实在的问题——不是概念不懂，而是 **产品怎么活过冷启动**。

这篇把群聊里最有代表性的问答整理成文，方便没跟课的读者也能看懂。

开源 Demo：[text2sql-mvp](https://github.com/xiexiaojing/text2sql-mvp)（支付场景） / [community-text2sql-mvp](https://github.com/xiexiaojing/community-text2sql-mvp)（社区场景）。克隆后执行 start.sh，本地 8777 端口体验。


## 一、快速模式到底什么时候调 LLM？

有同学试了一句：「微信的订单有多少条」，系统生成了 **全量 COUNT** 的 SQL，感觉数据不对。问：**未命中为什么不走 LLM？**

### 1. 这不是未命中，是误命中

在支付 Demo 里，这句话命中了 `payment_order_count`（订单总数），因为问句里同时有「订单」和「有多少」——规则打分通过了，直接套全量计数模板，**不会再去问 LLM**。

快速模式的设计是：

```text
高置信命中 executable 意图 → 渲染 SQL 模板 → 执行
未命中 / 不可执行 → 拒答（不让 LLM 自由写 SQL）
LLM 只在：多个意图定不下来（精排+抽槽），或 Schema 兜底路径
```

所以问题不在「LLM 不够多」，而在 **意图边界没配好**——带「微信/渠道」限定的问题，应单独配「按渠道查订单数」意图，并把 `priority` 设高一点，避免被总数意图抢走；或者在总数意图的 `match` 里加 **排除词**（`none`），问句一旦出现这些词就不再命中该意图，从而 **宁可拒答，不可错答**：

```yaml
- id: payment_order_count
  match:
    all: ["订单"]
    any: ["总数", "有多少", "数量", "多少", "几笔", "笔数"]
    none: ["微信", "支付宝", "银联", "渠道"]
```

### 2. 上线产品里，错答比不能答致命

新产品预设场景不全，用户问法又散——**误命中的杀伤力远大于未命中**。未命中还能说「这个口径还没配」；错答一次，用户觉得「系统数据不可信」，前期口碑就很难挽回。

补场景确实是慢工、有门槛，但 **防误命中和补新意图一样，都是 P0**。


## 二、场景不够，能不能前期先 LLM，后期再切模板？

群里共识：**认同模板在效率、成本、口径稳定上的价值，80% 高频场景模板够用；难的是配好之前的那一段。**

有人提议：前期让 LLM 跑场景和数据，慢慢补足后再切换。

### 方向对，但要拆两条路

| 路径 | 用途 | 对谁 |
|------|------|------|
| **LLM 探路**（内测/运营） | 收真实问法、验 SQL、沉淀 yaml | 对内 |
| **快速模式**（模板） | 只交付已审核口径 | 对外 |

**不要把「LLM 探路」和「对外交付」混在同一条链路。** LLM 前期用来 **发现场景、验证口径**；每跑通一个，固化成意图 + 模板 + eval_cases。切换的不是「某天关掉 LLM」，而是 **高频进模板，长尾留探索**。

### 新产品三阶段（可直接当路线图）

**第一阶段：窄域 + 拒答/追问**

- 对外只承诺 5～8 个高频口径（界面、文档写清楚）
- 带限定词但没专门意图的 → 追问或拒答（如物业「负责人」要问清是经理还是客服）
- 目标：**零错答**，可以接受「很多问题还不能问」

**第二阶段：LLM 探索 + 沉淀**

- 内测开「探索模式」，结果标「需人工核实」
- 每周把验证通过的问法沉淀为配置 + 回归测试

**第三阶段：模板为主，LLM 收长尾**

- 高频走快速模式；复杂报告走专家模式


## 三、冷启动可以「反着来」：先场景，后规则

群里有一条很务实的路径：

> 先去拿 **真实业务场景和数据**，针对特定口径先出 **规则 + 模板**；说法多变、规则盖不住的，再用 **轻量语义层**（向量召回、关键词扩展等）补一层；先出一版说得过去的效果。**LLM 兜底用。**

这和课上 **「先规则、后模型」** 完全一致：

```text
用户问题
  → 规则 + 模板（高频、口径明确，主路径）
  → 语义匹配层（说法多变、仍在已配置场景内）
  → LLM 兜底（前面都搞不定）
```

| 层级 | 适合什么 |
|------|----------|
| **规则 + SQL 模板** | 口径固定、可审计、可回归 |
| **语义匹配层** | 降低「每条问法手写规则」的成本；仍需场景定义，不是免配口径 |
| **LLM 兜底** | 长尾；必须有 Harness（白名单、租户注入、只读） |

**先场景、后规则**——比从空白猜问法务实得多，冷启动推荐这条路。

后续麻烦在于：**场景扩大、数据增多，人跟不上。** 缓解靠：

- 审计日志驱动：只补 TOP N 未命中/误命中
- 规则能盖的不往上堆模型
- eval 自动化：改一条配置必跑回归
- 产品分层：「已验收口径」vs「AI 探索」分开标


## 四、Multi-Agent 业内有好方案吗？

有同学做过流程图式 Multi-Agent，感觉不实用——**这个感受非常典型**。

业内 **没有「开箱即用、产品级稳」的 Multi-Agent 银弹**。Demo 层框架很多；产品层多数成功案例其实是 **「看起来像 Agent，本质是受控编排」**。

| 做法 | 问题 |
|------|------|
| 纯流程图 Multi-Agent | 太 rigid，维护爆炸，错误难恢复 |
| Agent 自由群聊 | 慢、贵、误差乘法累积 |

相对靠谱的路径：

- **Router 分诊 + 单链路**（80% 场景，智能问数快速模式就是这种）
- 必须多角色时：**Supervisor + 窄职责 Worker**，结构化 JSON 进出
- **Harness 管流程，LLM 只补模糊处**

智能问数先把快速模式（检索 + 模板）做稳，再碰 A2A / Multi-Agent。

另：**国内模型做 Multi-Agent 产品化，目前性价比往往不高**——不是完全不能用，而是 Demo 能跑、上线后慢、贵、难维护。不等于必须顶配模型；瓶颈常在 **编排和护栏**，不在换一个更大的模型。


## 五、几条可以写进 README 的产品原则

**第一，宁可拒答/追问，不可错答。** 信任一次翻车很难补。

**第二，冷启动可以反着来。** 先真实场景，再规则 + 语义层 + LLM 兜底；别对外默认「随便问」。

**第三，LLM 分对内对外。** 对内探路、沉淀配置；对外只交付已验收的快速模式。

**第四，补场景靠日志，不靠空想。** unsupported / 误命中 TOP 列表 + 业务方给的典型问题，比团队脑补高效。

**第五，模板是终态，不是 Day 1 的唯一手段。** 80% 稳了再上模板扛 SLA；之前用探索模式诚实交付。

**第六，Multi-Agent 后置。** Router + 模板 + 工具链往往够用；Agent 越多，不确定性是乘法。


## 六、结语

群里的讨论其实绕不开一个核心：

**工程上我们要的是「准、快、可审计」；产品冷启动要的是「先活下来、别错答」。**

两者不矛盾——用 **规则 + 模板** 扛已验收场景，用 **语义匹配 + LLM** 做泛化与兜底，用 **日志与 eval** 把枯燥的补场景变成可迭代的流水线。

模板匹配效率好、成本低，这个判断没错；难的是 **配好之前别用错答换覆盖**。把边界讲清楚，比假装「什么都能问」更诚实。

有问题欢迎继续在群里抛 **具体问句 + 期望 SQL**——问得越具体，越好配意图。
